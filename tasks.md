🔐 PHASE 2 — BILLION DOLLAR SECURITY FORTRESS AUDIT

Security is not a feature. Security is the foundation. One breach on a $1 Billion platform destroys everything. Find every single vulnerability. Leave no attack surface unexamined.

Injection Attack Surface

SQL Injection — Every database call must use parameterized queries via Supabase client. Zero raw SQL from user input. Zero string concatenation in queries. Audit every single DB call
NoSQL Injection — All document queries validated and sanitized. No user input passed directly to query operators
Command Injection — Zero shell execution from user-controlled data. Strict allowlists for any system commands
LDAP Injection — All LDAP special characters escaped before any directory queries
XXE Injection — External entity processing disabled in all XML parsers. DTD processing disabled
Server-Side Template Injection — User input never reaches template engine evaluation. Templates are static with safe variable substitution only
GraphQL Injection — Query depth limited, query cost analyzed, all inputs validated, introspection disabled in production
Email Header Injection — All email headers sanitized, no user input in CC/BCC/From fields
Log Injection — User input sanitized before logging, no raw user strings in log outputs
ReDoS — All regular expressions audited for catastrophic backtracking

Web Application Attack Surface

XSS (Stored, Reflected, DOM) — DOMPurify on all user-rendered content, strict Content Security Policy blocking inline scripts, HttpOnly cookies preventing JS access, output encoding on all user data
CSRF — SameSite=Strict cookies, anti-CSRF tokens on all state-changing requests, origin validation on all API routes
Clickjacking — X-Frame-Options: DENY on all pages, frame-ancestors 'none' in CSP
CORS — Zero wildcard origins in production, explicit origin whitelist, credentials flag handled correctly
Open Redirect — All redirects server-validated against allowlist, no user-controlled redirect destinations
IDOR — Supabase Row Level Security on every table, object-level authorization checked on every API route before returning or modifying data
Mass Assignment — Only explicitly allowed fields accepted from user input, no spreading req.body directly
Path Traversal — No user input in file paths, all paths normalized and validated against allowed directories
HTTP Request Smuggling — Consistent Content-Length headers, server configured to reject ambiguous requests
Prototype Pollution — Object.create(null) for user-data objects, JSON schema validation on all inputs

Authentication & Session Security

MFA enforced for all admin and teacher accounts
JWT security — strong secrets (256-bit minimum), short expiry, refresh token rotation, algorithm locked to HS256/RS256
Session management — absolute timeout, idle timeout, concurrent session limits, secure invalidation on logout
Password security — bcrypt with cost factor 12+, breach password checking, complexity requirements
Account enumeration — same response time and message for existing vs non-existing accounts
Brute force protection — exponential backoff, account lockout after 5 attempts, CAPTCHA after 3 attempts
OAuth security — state parameter validated, PKCE implemented, redirect URIs strictly validated

Data Security & Privacy

Encryption at rest — AES-256 for all stored data via Supabase
Encryption in transit — TLS 1.3 minimum, HSTS with preloading, no HTTP allowed
Field-level encryption — All PII encrypted at application level before storage
Data masking — Sensitive data masked in all logs, error messages, API responses
Secure file uploads — MIME type validation (not just extension), file size limits enforced, malware scanning via ClamAV API, files served from separate domain
Secret management — All secrets in Vercel environment variables, Supabase Vault for DB secrets, zero secrets in code or git history
Data minimization — Only collect data that is necessary, auto-purge old data per retention policy

Infrastructure Security

Cloudflare WAF — Rules blocking OWASP Top 10, rate limiting at edge, bot protection enabled
Vercel security headers — CSP, HSTS, X-Content-Type-Options, Referrer-Policy, Permissions-Policy all configured
Supabase RLS — Every single table has Row Level Security enabled and tested. Users can ONLY access their own school's data
API security — Every route validates session, Zod validates all inputs, responses never expose internal errors, rate limited per user and IP via Upstash Redis
Dependency security — npm audit clean, Snyk scanning in CI/CD, Dependabot alerts enabled, no packages with known critical CVEs
CI/CD security — No secrets in code, branch protection on main, required PR reviews, automated security scanning on every push
